## 1. Quickstart + demo NEWS + 截图

### 为什么重要

当前项目已经具备产品雏形，但第一次上手仍依赖以下前提：

- 用户知道什么是 `agent` / Cursor CLI
- 用户能先生成一批 NEWS 文件
- 用户能理解空页面是“还没生成数据”，不是“项目坏了”

对于开源项目，这是典型流失点。优秀的开源工具应该让陌生人 clone 后在几分钟内看到“可工作的结果”，而不是先读半天说明再补数据。

### 目标

把首次体验改成：

1. `pnpm install`
2. `pnpm dev`
3. 打开页面就能看到一套可浏览的 demo 数据
4. 再决定要不要配置自己的生成链路

### 建议实现细节

#### 1.1 Demo 数据目录

建议新增一个可提交到仓库的 demo 目录，例如：

```text
demo-news/
  general/
  finance/
  ai-tech/
  science/
  crypto/
```

要求：

- 每个主题至少 1 份样例
- 内容可精简，但结构必须符合真实 Markdown 约定
- 文件名和真实 NEWS 格式一致
- 内容不要依赖真实的近期新闻，避免时间过期感太强

建议用途：

- 本地未配置 `NEWS/` 时，可选读取 demo 数据
- 截图、测试、文档说明都基于这套样例

#### 1.2 Demo 模式

建议支持至少一种 demo 使用方式：

- 方式 A：环境变量 `NEWS_ROOT=demo-news`
- 方式 B：如果 `NEWS/` 为空，则自动 fallback 到 `demo-news/`
- 方式 C：README 中明确提供“复制 demo 到 NEWS”的命令

推荐优先：

- A + C

原因：

- 行为更明确
- 不会让真实用户误以为 demo 就是自己的数据

#### 1.3 Quickstart 文档结构

README 顶部应出现真正的 Quickstart，而不是先讲理念。

建议顺序：

1. 项目一句话介绍
2. 截图/GIF
3. 3 步跑起来
4. 再讲原理和架构

建议内容：

```md
## Quickstart

1. 安装依赖
2. 使用 demo 数据启动
3. 打开 localhost
4. 浏览首页 / About / Runtime
```

#### 1.4 截图与 GIF

建议新增：

- 首页截图
- 日报正文截图
- Runtime 页截图
- 10 秒以内短 GIF：打开首页 -> 点击某条日报 -> 回到首页搜索

要求：

- 使用稳定 demo 数据生成
- 保持亮色为主，暗色可作为补充
- 截图文件名明确，如：
  - `public/screenshots/home-light.png`
  - `public/screenshots/article.png`
  - `public/screenshots/runtime.png`

### MVP

- 一套可提交仓库的 demo NEWS
- README 顶部 Quickstart
- 至少 2 张截图
- 本地能明确使用 demo 数据启动

### 验收标准

- 新用户不需要先生成新闻，也能看到完整 UI
- README 前 2 分钟内能让人知道如何跑起来
- 项目首页截图与实际页面一致

## 2. 运行状态 / 最近成功时间 / 失败反馈

### 为什么重要

如果 S-News 要承担“每日使用”的角色，生成链路就不能只有一个“点一下运行”的按钮。用户真正关心的是：

- 现在是否正在生成
- 上一次什么时候成功
- 失败了没有
- 失败原因是什么
- 是否生成了新文件

没有这些反馈，Runtime 只像一个“会转圈的按钮”，不够像可靠工具。

### 目标

把 Runtime 从“触发器”升级成“运行面板”。

### 建议实现细节

#### 2.1 运行状态模型

建议区分以下状态：

- `idle`：尚未运行
- `running`：正在执行
- `success`：执行成功
- `error`：执行失败
- `partial`：部分成功（未来支持 all 主题时可用）

#### 2.2 最近运行记录

建议至少保存最近一次运行结果：

- 主题
- 开始时间
- 结束时间
- 耗时
- 状态
- 标准输出摘要
- 标准错误摘要
- 是否检测到新文件

存储方式建议：

- 初期：写入本地 `runtime-state.json`
- 或由 API 在内存中维护并返回

更推荐：

- 文件落盘

因为：

- 页面刷新后还能看到
- 更适合本地工具定位

#### 2.3 最近成功时间

首页或 Runtime 中建议展示：

- 最近一次成功生成时间
- 最近一次生成的主题
- 最近一次新增日报的日期

这是“这工具有没有活着”的最直接信号。

#### 2.4 错误反馈层级

不要直接把整段 stderr 倾倒给普通用户。

建议分三层：

1. 用户可读摘要
   - 例如：`agent 未安装`
   - 例如：`脚本执行超时`
   - 例如：`生成完成但未检测到新文件`

2. 关键诊断信息
   - exit code
   - stderr 前 10 行
   - stdout 最后 10 行

3. 完整日志（可折叠）

#### 2.5 空跑与成功判断

“命令执行完成”不应等于“生成成功”。

建议成功判定包含：

- 进程退出码为 0
- 目标目录有新文件或目标日期文件被更新

否则显示：

- `执行完成，但没有检测到新日报`

### MVP

- Runtime 页显示最近一次运行时间、结果、耗时
- 错误信息分摘要和详情
- 成功后显示是否生成新日报

### 验收标准

- 用户刷新页面后仍能看到最近状态
- 用户能看懂“失败在哪”
- 成功与“只是不报错”严格区分

## 3. 今日简报 / Top N

### 为什么重要

现在首页已经能看“今天有哪些日报”，但还差一个“今天先看什么”的层。优秀的信息工具不只存东西，还应该在 10 秒内帮用户决定阅读顺序。

### 目标

新增一块 `今日简报` 或 `Top N`，作为跨主题的“头版导读”。

### 建议实现细节

#### 3.1 数据来源

优先复用现有解析结果：

- `latestEntries`
- `entry.highlights`
- `entry.takeaway`
- `entry.title`
- `entry.description`

初版不需要额外 LLM 总结，不新增生成成本。

#### 3.2 生成规则（推荐初版）

每个当日主题取：

- `highlights[0]` 优先
- 若无 `highlights`，取 `takeaway`
- 再无则取 `description`

形成一组跨主题头条。

排序建议：

- 按主题固定顺序
- 或未来支持手动优先级

#### 3.3 展示形态

建议不要做成又一排普通卡片。

更好的形式：

- 头版导读条目
- 左侧主题徽记
- 中间一句导读
- 右侧跳转按钮

文案风格：

- 像报纸导读，不像“智能推荐”

例如：

- `金融：市场今天先盯通胀数据和科技股财报`
- `AI/Tech：模型发布很热闹，但真正值得看的是监管变化`

#### 3.4 数量控制

建议：

- 默认展示 3 到 5 条
- 对应当天实际可读主题数

不要超过：

- 6 条

否则首页会重新变拥挤。

#### 3.5 空状态

若今天没有日报：

- 不显示该模块
- 或显示：`今天还没上桌，先去翻昨日头版`

### MVP

- 首页新增 `今日简报`
- 基于现有 `latestEntries` 生成
- 每主题取 1 条

### 验收标准

- 用户打开首页 10 秒内能知道“今天该先看什么”
- 不额外依赖生成链路
- 不引入新 API 或新数据格式

## 4. Markdown schema 与 fixture 测试

### 为什么重要

S-News 的真正数据层不是数据库，而是 Markdown。也就是说：

- Markdown 结构就是 schema
- 解析器就是数据兼容层

如果 schema 不清楚、fixture 不固定、测试不覆盖，后续一改 command 模板或增加主题，就很容易让页面悄悄坏掉。

### 目标

把 Markdown 从“约定俗成”提升为“半正式协议”。

### 建议实现细节

#### 4.1 明确定义 schema

建议单独写一节或单独文件，明确以下字段：

- 文件名规则
- 一级标题规则
- 引用块摘要规则
- 二级标题规则
- 三级标题规则
- `**摘要：**`
- `**链接：**`
- `**简评：**`
- `今日小结`
- `总体定性 / 今日定性`

需要明确：

- 哪些是必须字段
- 哪些缺失时允许 fallback
- 哪些只影响增强展示，不影响主流程

#### 4.2 Fixture 样例

建议在仓库中提供固定 fixture：

```text
tests/fixtures/news/
  general/2026-03-01_xxx.md
  finance/2026-03-01_xxx.md
  ...
```

这些样例应该覆盖：

- 标准完整日报
- 缺少 highlights 的日报
- 缺少 takeaway 的日报
- 标题/摘要异常格式
- 空目录

#### 4.3 解析测试

需要重点测：

- `extractTitle`
- `extractDescription`
- `extractTakeaway`
- `extractHighlights`
- `groupEntriesByDate`
- `searchEntries`

建议断言内容：

- 解析结果字段正确
- 缺失字段能正常 fallback
- 日期排序正确
- 搜索行为不因 topic 过滤出错

#### 4.4 路由级别测试

可逐步增加：

- 首页在 fixture 数据下能正常渲染
- 某日报正文页能正常渲染
- 空数据时展示空状态

### MVP

- 文档里明确 Markdown schema
- 至少 3 组 fixture
- 覆盖核心解析函数的单元测试

### 验收标准

- 改 Command 模板时，能快速知道是否破坏解析
- 空字段与异常格式不再只能靠手测发现
- demo 数据、fixture 数据和 README 示例保持一致

## 5. 主题扩展改成配置优先，而不是代码优先

### 为什么重要

如果加一个主题需要同时改：

- `lib/news-meta.ts`
- scripts
- 命令定义
- 首页文案
- About 文案
- 可能还有路由说明

那它更像一个个人项目，而不是开源模板。

优秀开源工具应该让“新增主题”尽量接近“增加一份配置 + 一份 command + 一份脚本”。

### 目标

把主题定义收敛成一个稳定配置层，界面与运行逻辑都尽量消费配置，而不是散落硬编码。

### 建议实现细节

#### 5.1 主题配置源

建议最终形成一个中心配置文件，例如：

```ts
topics.config.ts
```

每个主题至少包含：

- `key`
- `label`
- `shortLabel`
- `description`
- `folder`
- `accent`
- `commandPath`
- `scriptPath`
- `enabled`
- `order`
- 可选 `icon`

现有 `lib/news-meta.ts` 已经是这个方向，但还可以更彻底地成为唯一事实来源。

#### 5.2 页面消费配置

以下地方都应从配置自动读取：

- 首页筛选
- About 页 topic 列表
- Runtime 主题按钮
- 导航与统计数字
- topic icon 映射

目标：

- 增删主题后，页面自动跟着变

#### 5.3 脚本策略

长期看，脚本也应尽量配置驱动：

- `run_all_news.sh` 读取启用主题
- 或用 Node/Python 统一执行器，根据配置决定调用哪个 command

这样能减少：

- 每新增一个主题就复制一份 shell 脚本

#### 5.4 主题启用/禁用

建议支持：

- 已配置但暂不启用的主题

对开源非常重要，因为：

- 仓库可以预置很多主题模板
- 默认只展示稳定主题

#### 5.5 新主题接入文档

建议写成固定步骤：

1. 新增 command
2. 新增脚本或执行器配置
3. 在主题配置文件中注册
4. 放入一个样例 NEWS 文件
5. 运行检查

这样别人能真正二开，不需要读完整个代码库。

### MVP

- 明确一个唯一主题配置源
- 首页/About/Runtime 全部消费同一配置
- 补一节“如何新增主题”的文档

### 验收标准

- 新增一个主题不需要改动多个无关文件
- UI 与运行逻辑不会出现“这里有主题，那里没主题”的不同步
- 第三方开发者可以照文档新增主题

## 建议优先级

### 第一优先级

1. `Quickstart + demo NEWS + 截图`
2. `运行状态 / 最近成功时间 / 失败反馈`

原因：

- 这两项最直接影响“别人能不能用”和“你自己敢不敢长期用”

### 第二优先级

3. `Markdown schema 与 fixture 测试`
4. `今日简报 / Top N`

原因：

- 一个提升稳定性
- 一个提升每日打开价值

### 第三优先级

5. `主题扩展改成配置优先`

原因：

- 这是最像“开源模板化”的一步，但也最适合在前面几项稳定后推进

---

## 总结

这五件事分别对应五个维度：

- `Quickstart + demo + 截图`：降低上手门槛
- `运行状态 / 失败反馈`：提升工具可靠感
- `今日简报 / Top N`：增强每日使用价值
- `schema + fixture 测试`：提升系统稳定性
- `配置优先扩展主题`：增强开源可复用性

如果这五项能逐步落地，S-News 会明显从“一个漂亮的本地新闻阅读器”进化成“一个真正可持续使用、可协作、可二开的开源工具”。
